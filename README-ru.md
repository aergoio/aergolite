<h1 align="center">Бездоверительная репликация SQLite</h1>

AergoLite предоставляет реплицированную базу данных SQLite, защищенную приватным и легковесным блокчейном.

Каждое приложение имеет локальную реплику базы данных.

Новые транзакции базы данных распределяются между всеми узлами, и после достижения консенсуса относительно порядка выполнения все узлы выполняют транзакции. Поскольку порядок выполнения одинаков, все узлы имеют одинаковое содержимое базы данных.

Приложения также могут записывать в локальную базу данных, когда они находятся в автономном режиме. Транзакции базы данных сохраняются в локальной очереди и отправляются в сеть после восстановления подключения. Приложение будет читать новое состояние базы данных с автономными изменениями и может проверить, были ли автономные транзакции обработаны глобальным консенсусом. Если транзакции отклонены, база данных вернется к предыдущему состоянию.

AergoLite использует **специальную технологию блокчейна**, ориентированную на **устройства с ограниченными ресурсами**.

Протокол консенсуса использует **Верифицируемую Случайную Функцию (VRF)** для определения того, какой узел создаст следующий блок, и узлы не могут заранее узнать, какой узел выбран. Это обеспечивает защиту от атак типа "отказ в обслуживании" (DoS).

AergoLite использует **абсолютную финальность**. После того, как узлы достигают консенсуса относительно нового блока и транзакции подтверждены, возврата нет. Также нет необходимости создавать новые блоки, если нет транзакций для обработки (в отличие от вероятностной финальности).

Для проверки целостности блокчейна и состояния базы данных требуется только последний блок, поэтому узлам не нужно хранить и проверять всю историю блоков и транзакций.
Также возможно настроить некоторые узлы для хранения всей истории в целях аудита.

Он также использует **хэш состояния базы данных**. Это позволяет узлам проверить, имеют ли они точно такое же содержимое в базе данных, защищает от намеренных изменений файла базы данных, а также работает как проверка целостности для обнаружения сбоев на носителе данных.

Этот финальный хэш обновляется только с использованием измененных страниц в каждом новом блоке. Не нужно загружать всю базу данных для расчета нового состояния. Проверка целостности также выполняется только при загрузке новой страницы базы данных. Это значительно повышает производительность базы данных.

Полученное решение не требует большого дискового пространства, использует мало процессорного времени и оперативной памяти.

Сетевой трафик также легковесный для снижения энергопотребления. Новые пакеты передаются только при наличии новых транзакций базы данных.

Эта технология позволяет запускать приватный блокчейн на устройствах IoT и мобильных устройствах.

Вам не нужно знать, как работает блокчейн, чтобы использовать его.

## Использование

Скомпилированная библиотека поддерживает как нативные файлы баз данных SQLite, так и базы данных SQLite с поддержкой блокчейна, поэтому приложение может открывать нативные базы данных SQLite и базы данных с блокчейном одновременно.

Библиотека работает точно так же для обычной базы данных SQLite.

Для открытия базы данных с поддержкой блокчейна мы используем параметр URI: `blockchain`

Пример:

```
"file:test.db?blockchain=on"
```


## Администратор блокчейна

AergoLite реализует приватный блокчейн. Это означает, что вы или ваша организация можете иметь
свой собственный приватный блокчейн(ы), в котором вы контролируете, что может происходить.

Сущность, которая имеет контроль над блокчейном, называется администратором блокчейна. Это
пользователь, который имеет свою собственную пару приватного + публичного ключа.

Администратор блокчейна может:

* Добавлять узлы в сеть блокчейна
* Выполнять любые SQL команды
* Создавать хранимые процедуры

В будущих версиях он также сможет:

* Удалять узлы из сети блокчейна
* Добавлять пользователей в сеть

Вам нужно будет указать публичный ключ администратора блокчейна на каждом участвующем узле.

Это гарантирует, что:

1. Узлы могут проверить, что полученные команды приходят от администратора блокчейна
2. Узел не присоединится к сети, которая вам не принадлежит

Это делается через параметр `admin`, где публичный ключ может быть в нативном или hex формате.

Пример:

```
"file:test.db?blockchain=on&admin=95F9AB75CA1..."
```


## Безопасность

Базовые решения репликации позволяют узлам выполнять любые SQL команды в базе данных.
Они небезопасны, потому что злоумышленник, получивший контроль над одним узлом, может удалить и/или
перезаписать данные во всей сети.

Настоящее бездоверительное решение репликации должно ограничивать то, что могут делать узлы, и не доверять им.

Злоумышленник может быть способен изменить локальную базу данных, но это не отразится на других узлах.

Успешная атака на сеть потребует от злоумышленника получить контроль над большинством узлов. Этот дизайн обеспечивает надежный уровень безопасности, делая сложным для любой отдельной сущности скомпрометировать целостность системы.


## Хранимые процедуры

AergoLite использует хранимые процедуры, написанные на SQL, для контроля того, что могут делать узлы.

По соображениям безопасности:

1. Только администратор может создавать хранимые процедуры.
2. Узлы могут только выполнять эти хранимые процедуры. Другие SQL команды заблокированы.

Таким образом, они работают аналогично смарт-контрактам.

Вот пример:

```sql
CREATE PROCEDURE add_new_sale(@products) BEGIN
 -- убедиться, что вызывающий может вызвать эту функцию
 ASSERT txn_sender() IN (SELECT pubkey FROM authorizations WHERE type = 'sale');
 -- вставить новую продажу
 INSERT INTO sales (time) VALUES (datetime('now'));
 -- получить id продажи и сохранить его в переменной
 SET @sale_id = last_insert_rowid();
 -- вставить каждый продукт, используя ссылку на id продажи
 FOREACH @prod_id, @qty, @price IN @products DO
   INSERT INTO sale_items (sale_id, prod_id, qty, price) VALUES (@sale_id, @prod_id, @qty, @price);
 END LOOP;
 -- вернуть id продажи
 RETURN @sale_id;
END;
```

Это можно вызвать из SQL следующим образом:

```sql
CALL add_new_sale( [['123', 1, 10.00],['456', 2, 20.00]] );
```

Полная справка по доступным командам может быть найдена [здесь](https://github.com/aergoio/sqlite-stored-procedures)

Возможно иметь приложения, созданные с использованием разных языков программирования в одной сети, так как основная логика выполняется с помощью хранимых процедур.


## Неизменяемость

Если вы ограничите хранимые процедуры только командами `INSERT INTO`, то содержимое базы данных будет неизменяемым.

Но даже при обычном использовании, позволяющем обновление и удаление строк, история изменений (все SQL команды) записывается в легковесный блокчейн и не может быть удалена, что позволяет как восстановление данных, так и аудит.


## Защита приватного ключа

Каждый узел генерирует уникальную пару приватного + публичного ключа. Они идентифицируются и авторизуются через свой публичный ключ.

На данный момент приватный ключ для каждого узла хранится в зашифрованном виде на устройстве. В будущих версиях может поддерживаться защита приватного ключа на основе аппаратного обеспечения.

Нам нужно указать пароль, используемый для расшифровки приватного ключа, используя параметр URI `password`.

Пример:

```
"file:test.db?blockchain=on&admin=95F9AB75CA1...&password=testing"
```

Пароль может быть разным на каждом узле.

Опционально ваше приложение может быть ответственным за генерацию и хранение приватного ключа для каждого узла. В этом случае оно может указать приватный ключ в hex формате библиотеке через URI, используя параметр `privkey`:

```
"file:test.db?blockchain=on&admin=95F9AB75CA1...&privkey=AABBCCDD..."
```

Это может быть полезно при использовании AergoLite в контейнерах, где каждый экземпляр должен иметь другой приватный ключ.

Администратор блокчейна отвечает за безопасное хранение своего приватного ключа. Мы рекомендуем не хранить его на одном из узлов блокчейна и не в открытом формате. Он должен быть зашифрован и храниться на внешнем устройстве или носителе. Бумажный кошелек также является хорошей идеей. Лучший вариант - использовать аппаратный кошелек.

## Обнаружение узлов

Узлу нужно обнаружить своих пиров в сети блокчейна.

Мы указываем метод обнаружения узлов, используя параметр URI `discovery`.

Есть 2 варианта обнаружения узлов:

### 1. Локальная UDP широковещательная рассылка

Этот метод отправляет UDP широковещательный пакет в локальной сети на указанный порт.

Все узлы из той же локальной сети должны использовать тот же номер порта.

Пример:

```
"file:test.db?blockchain=on&discovery=local:4329"
```

### 2. Известные узлы

При этом методе некоторые узлы имеют фиксированный IP-адрес, а другие узлы подключаются к ним.

Узлы с известным адресом также должны привязаться к определенному TCP порту. Это указывается с помощью параметра `bind`.

Пример URI для "известного" узла:

```
"file:test.db?blockchain=on&bind=5501"
```

Другие узлы должны иметь явный параметр `discovery`, содержащий адрес известных узлов.

Пример URI для других узлов:

```
"file:test.db?blockchain=on&discovery=<ip-address>:<port>"
```

Мы также можем указать адреса нескольких известных узлов:

```
"file:test.db?blockchain=on&discovery=<ip-address1>:<port1>,<ip-address2>:<port2>"
```

После установления соединения и принятия узла они обмениваются списком адресов активных узлов.

### 3. Смешивание обоих методов

Мы также можем использовать оба вышеуказанных метода одновременно. Это может быть полезно, когда у нас есть некоторые узлы в локальной сети и другие, которые находятся снаружи.

Мы можем зафиксировать адрес одного или нескольких узлов, чтобы они могли быть найдены узлами извне локальной сети.

Узлы в локальной сети будут обнаруживать локальные узлы через UDP широковещательную рассылку и могут либо подключаться к известным узлам вне локальной сети, либо принимать соединения от них.

Известные узлы могут привязаться к порту, найти локальные узлы через широковещательную рассылку, а также подключиться к внешним известным узлам. Пример:

```
"file:test.db?blockchain=on&bind=1234&discovery=local:1234,<outside_ip1>:<port1>,<outside_ip2>:<port2>"
```

Узлы без фиксированного адреса будут использовать локальное обнаружение и подключение к внешним известным узлам:

```
"file:test.db?blockchain=on&discovery=local:1234,<outside_ip1>:<port1>,<outside_ip2>:<port2>"
```

Если узлы в этой локальной сети только принимают соединения извне, то параметр `discovery` должен содержать только метод локального обнаружения.


## Список подключенных узлов

Вы можете получить список узлов в вашей приватной сети блокчейна, используя команду:

```
PRAGMA nodes
```

Она выведет все авторизованные узлы, подключенные или нет, а также подключенные узлы, которые еще не авторизованы.


## Добавление узлов в сеть

После получения списка подключенных узлов с помощью вышеуказанной команды администратор блокчейна может авторизовать узлы, используя команду:

```
PRAGMA add_node="<public key>"
```

Только администратор блокчейна может добавлять узлы в сеть.

Первый узел, который должен быть авторизован, должен быть тем, на котором выполняется команда.

Авторизации для других узлов должны выполняться на узлах, которые уже авторизованы.

Вышеуказанная команда будет отправлена на устройство Ledger для подписания, если устройство подключено, в противном случае она вызовет callback подписания транзакции, где транзакция должна быть подписана с использованием приватного ключа администратора блокчейна.


## Указание типа узла

По умолчанию узел авторизуется как **легкий** узел (не хранит историю блоков). Чтобы авторизовать его как **полный** узел, добавьте `full:` перед публичным ключом узла:

```
PRAGMA add_node="full:<public key>"
```

Чтобы изменить тип узла после того, как он уже был авторизован, используйте команду `node_type`. Она имеет следующий формат:

```
PRAGMA node_type="<type>:<nodes>"
```

Тип может быть `full` или `light`. "nodes" - это список идентификаторов узлов (публичный ключ или id узла), разделенных запятыми, или `*` для всех авторизованных узлов.

Вот несколько примеров:

```
PRAGMA node_type="full:Am12..abc1"
PRAGMA node_type="full:Am12..abc1,Am12..abc2,Am12..abc3"
PRAGMA node_type="full:1287649477,3817592406,2373041549"
PRAGMA node_type="full:*"
PRAGMA node_type="light:1287649477"
```


## Подписание транзакций

В AergoLite транзакции блокчейна строятся с использованием SQL команд из транзакций базы данных.

Каждая транзакция базы данных генерирует одну транзакцию блокчейна.

Эти транзакции должны быть подписаны, чтобы быть принятыми сетью и включенными в блокчейн.

Две сущности могут подписывать транзакции:

* Администратор
* Каждый авторизованный узел

Если транзакция требует специальных прав, библиотека AergoLite отправит ее для подписания администратором. В противном случае она автоматически подпишет ее, используя приватный ключ узла.

Если в вашей сети не используется устройство Ledger, то хотя бы один узел должен зарегистрировать функцию, которая будет использоваться для подписания транзакций от администратора

Пример на Python:

```python
def on_sign_transaction(data):
  print "txn to be signed: " + data
  signature = sign(data, privkey)
  return hex(pubkey) + ":" + hex(signature)

con.create_function("sign_transaction", 1, on_sign_transaction)
```

> **ВНИМАНИЕ:** Функция обратного вызова вызывается **рабочим потоком**!!
> Ваше приложение должно подписать транзакцию и вернуться как можно быстрее!

Если специальная команда, требующая привилегий администратора, выполняется на узле, но не подписана им, то транзакция будет отклонена.


## Получение статуса

Есть 2 способа получить статус:

1. Локально через команды PRAGMA
2. Удаленно, отправляя запросы статуса через UDP пакеты


### Статус базы данных

Ваше приложение должно проверить, готова ли локальная база данных для чтения и записи, прежде чем выполнять любую SQL команду.

Эта проверка выполняется командой:

```
PRAGMA db_is_ready
```

Она возвращает `1`, когда приложение может читать из базы данных и записывать в нее, в противном случае возвращает `0`.


### Статус блокчейна

Это содержит информацию о локальном блокчейне, локальной базе данных и сети.

Его можно запросить локально, используя команду:

```
PRAGMA blockchain_status
```

Она вернет результат в формате JSON, подобный следующему:

```

  "use_blockchain": true,

    "integrity": {
      "state": "OK",
      "chain": "pruned"
    }
  },

  "node": {
    "id": 1366464921,
    "pubkey": "AmNdtXoBk6mYwgq2XDsx8pW9cvmoTQ3bp7v7kJxBcckvrC8HWBrE",

  "upstream_state": "in sync"
```

### Статус мемпула

Он возвращает ожидающие транзакции в локальном мемпуле.

```
PRAGMA mempool
```

Он вернет результат в формате JSON, подобный следующему:

```
[{
  "id": 17698765927658,
  "node_id": 123,
  "nonce": 18,
  "timestamp": "2020-11-30 09:55:13",
  "commands": [
    "INSERT INTO test VALUES ('hello world!')"
  ]
}]
```

### Информация об узле, определенная приложением

Ваше приложение может установить информацию, специфичную для узла, используя эту команду:

```
PRAGMA node_info=<text>
```

Текстовое значение может быть одним идентификатором узла или может содержать много информации, сериализованной в любом текстовом формате. Только ваши приложения будут использовать это.

Эта информация хранится в памяти локально и также отправляется подключенным пирам. Она не сохраняется в базе данных и является динамической: при следующем выполнении этой команды с другим значением оно заменит предыдущее.

Последнее установленное значение для этого узла может быть получено локально с помощью команды `PRAGMA node_info`.

Возможно просмотреть значения от подключенных узлов в результате команды `PRAGMA nodes` в поле `extra`.


### Последний nonce

Каждая сгенерированная транзакция на конкретном узле имеет уникальный инкрементальный nonce.

Можно получить последний nonce текущего узла командой:

```
PRAGMA last_nonce
```

Если возвращаемое число равно нулю, это означает, что этот узел еще не генерировал транзакций.


### Статус транзакции

Для получения статуса локальной транзакции:

```
PRAGMA transaction_status(<nonce>)
```

Где `<nonce>` должен быть заменен на nonce транзакции. например: `PRAGMA transaction_status(3)`

Она вернет

На полных узлах:

* `unprocessed`: транзакция еще не обработана сетью
* `included`: достигнут консенсус, и транзакция была включена в блок
* `rejected`: достигнут консенсус, и транзакция была отклонена

На легких узлах:

* `unprocessed`: транзакция еще не обработана сетью
* `processed`: транзакция была обработана сетью, и был достигнут консенсус относительно ее результата

Легкие узлы не хранят информацию о конкретных транзакциях.


### Уведомление о транзакции

Чтобы быть проинформированным о том, была ли конкретная транзакция включена в блок или отклонена, приложение должно использовать функцию обратного вызова. Она настраивается как `пользовательская функция`:

Пример на Python:

```python
def on_processed_transaction(nonce, status):
  print "transaction " + str(nonce) + ": " + status
  return None

con.create_function("transaction_notification", 2, on_processed_transaction)
```

Пример на C:

```C
static void on_processed_transaction(sqlite3_context *context, int argc, sqlite3_value **argv){
  long long int nonce = sqlite3_value_int64(argv[0]);
  char *status = sqlite3_value_text(argv[1]);

  printf("transaction %lld: %s\n", nonce, status);

  sqlite3_result_null(context);
}

sqlite3_create_function(db, "transaction_notification", 2, SQLITE_UTF8 | SQLITE_DETERMINISTIC,
  NULL, &on_processed_transaction, NULL, NULL);
```

> **ВНИМАНИЕ:** Функция обратного вызова вызывается **рабочим потоком**!!
> Ваше приложение не должно использовать соединение с базой данных там и должно вернуться как можно быстрее!
> Оно может отправить уведомление в основной поток перед возвратом


### Уведомление об обновлении

Ваше приложение может быть проинформировано всякий раз, когда происходит обновление локальной базы данных из-за получения нового блока в блокчейне.

Уведомление выполняется с помощью функции обратного вызова, которая устанавливается с использованием `пользовательской функции`:

Пример на Python:

```python
def on_db_update(arg):
  print "update received"
  return None

con.create_function("update_notification", 1, on_db_update)
```

Пример на C:

```C
static void on_db_update(sqlite3_context *context, int argc, sqlite3_value **argv){
  puts("update received");
  sqlite3_result_null(context);
}

sqlite3_create_function(db, "update_notification", 1, SQLITE_UTF8 | SQLITE_DETERMINISTIC,
  NULL, &on_db_update, NULL, NULL);
```

> **ВНИМАНИЕ:** Функция обратного вызова вызывается **рабочим потоком**!!
> Ваше приложение не должно использовать соединение с базой данных там и должно вернуться как можно быстрее!
> Оно может отправить уведомление в основной поток перед возвратом


## Интервал блоков

Блоки создаются случайно выбранными узлами в каждом раунде.

AergoLite не создает пустые блоки. Если нет транзакций для обработки, то блок не создается.

Таймер для создания нового блока активируется, когда транзакция прибывает на узлы (и таймер еще не активен).

Этот интервал таймаута может быть настроен через URI, используя параметр `block_interval`.

Значение интерпретируется как миллисекунды.

```
"file:test.db?blockchain=on&block_interval=1000"
```

Если интервал блоков не указан, то библиотека будет использовать значение по умолчанию 3 секунды.


## Ограничения

Эта версия использует полностью связанную сеть для связи между узлами. Она работает с до 200 узлами (проверено через автоматизированные тесты) и, вероятно, больше. В будущем она также может содержать протокол на основе gossip для поддержки еще большего количества узлов.

Только 1 соединение к каждому файлу базы данных. Если есть много приложений, которым нужно получить доступ к файлу базы данных, каждое приложение должно иметь свою собственную копию базы данных, настроенную как отдельный узел.

Нумерация строк в таблицах rowid (тех, которые используют целое число в качестве первичного ключа) отличается от SQLite. Первые 32 бита - это id узла, а оставшиеся 32 бита - последовательные на узел. Это также означает, что каждый узел может создать до 2^32 строк в каждой таблице rowid.

Как и в любой системе репликации с несколькими мастерами, могут возникать конфликты. Вся транзакция может быть прервана в некоторых случаях, поэтому учитывайте это. См. выше, как приложение может проверить статус транзакции.
